# AGENTS.md

이 문서는 CAU000 프로젝트에서 작업하는 코딩 에이전트가 따라야 할 기본 규칙입니다.
변경 전에는 현재 파일 구조와 실행 방식을 먼저 확인하고, 기존 템플릿 스타일을 유지하세요.

## 프로젝트 개요

- 업무 코드: `CAU000`
- 주요 언어: Python 3.11.9
- 주요 GUI 프레임워크: PySide6
- 패키징: PyInstaller/PyArmor 기반 Windows `.exe` 빌드, PyInstaller 기반 macOS `.app` 빌드
- 기본 진입점:
  - `main.py`: PySide6 GUI 실행 진입점
  - `main_unattended.py`: 무인/스케줄러 실행 진입점

## 작업 원칙

- `Common/`과 `Setup/` 폴더의 내용물은 수정 금지입니다.
- 업무 유틸은 `Function/`, 상수는 `Service/constants.py`에 둡니다.
- 신규 업무 로직은 `main.py` 또는 `main_unattended.py`에 길게 작성하지 말고, 가능한 경우 `Function/` 또는 업무 목적에 맞는 별도 모듈로 분리하세요.
- `main.py` 또는 `main_unattended.py` 파일은 실행 흐름, 입력값 처리, 스레드/예외 처리, 로그/메일 처리 중심으로 유지하세요.
- GUI 관련 변경은 `main.py`와 `Resource/main_windows_ui.py`, `Resource/main_macos_ui.py`의 관계를 함께 확인하세요.
- `Constants`의 `PROCESS_ID`, `PROCESS_NAME`, `APP_VERSION`은 빌드 산출물, 로그, 메일 제목에 영향을 줄 수 있으므로 변경 전 의도를 확인하세요.
- 실행 시 `sys._MEIPASS`, `sys.frozen`, `program_dir`, `__file__` 기준 경로를 고려하는 패키징 호환 코드가 있으므로 임의로 제거하지 마세요.
- `Resource/` 파일 참조는 현재 작업 디렉터리에만 의존하지 않도록 주의하고, PyInstaller 패키징 환경에서도 동작하는지 확인하세요.
- 민감 정보가 필요하면 환경 변수, 설정 파일, 사용자 입력 등으로 분리하는 방안을 우선 검토하세요.
- 경로 처리 시 `os.path` 또는 `pathlib`를 사용하고, 실행 파일 패키징 환경을 고려하세요.
- 특정 운영체제 전용 명령, 절대 경로, 경로 구분자에 의존하는 코드는 피하고 Windows/macOS 실행 환경을 함께 고려하세요.

## README 문서화 원칙

- 파일을 생성, 수정, 삭제하거나 이름을 변경할 때마다 `README.md`에 반영할 내용이 있는지 반드시 확인하세요.
- 프로젝트의 프로세스, 구조, 주요 파일의 역할이 달라진 경우 `README.md`의 **프로세스** 및 **프로젝트 구조**를 같은 작업에서 함께 수정하세요.
- 기능, GUI 동작, 로그 형식, 설정, 의존성, 실행 방법 또는 빌드 방법이 달라진 경우 관련 설명과 예시 명령을 현재 코드에 맞게 갱신하세요.
- README 수정 후에는 문서에 적힌 파일명, 경로, 명령과 동작이 실제 프로젝트 상태와 일치하는지 검증하세요.
- README에 영향을 주지 않는 내부 주석이나 서식 변경인 경우에도 문서와의 일치 여부를 확인하되, 의미 없는 문서 변경은 추가하지 마세요.

## RPA 자동화 원칙

- 운영계 실행이 필요한 경우 사용자에게 대상과 범위를 확인받으세요.
- 실제 게시, 댓글 작성, 메일 발송, 파일 업로드, 승인 등 외부 상태를 변경하는 작업은 테스트 중 실행하지 마세요.
- 게시, 댓글, 메일, 업로드처럼 중복 실행의 영향이 큰 작업은 실행 전에 기존 처리 여부를 확인하세요.
- 무인 실행의 중복 기동을 방지하고, 중단 후 재실행 시 이미 완료된 건을 다시 처리하지 않도록 설계하세요.
- 브라우저, WebDriver, 외부 프로세스 및 임시 파일은 성공 여부와 관계없이 정리되도록 `finally` 흐름을 유지하세요.
- 로그에는 업무 단계와 결과를 기록하되 비밀번호, 토큰, 쿠키, 인증 헤더, 개인 정보 및 원문 전체를 출력하지 마세요.
- 테스트에는 테스트 계정과 비운영 데이터를 사용하고, 가능하면 dry-run 또는 외부 호출 모킹 방식을 제공하세요.
- Chrome과 WebDriver의 버전 호환성을 확인하고, 드라이버 다운로드 또는 업데이트가 발생하는 경우 네트워크 및 배포 환경에 미치는 영향을 고려하세요.

## Selenium 웹 자동화 및 크롤링 원칙

- Google OAuth 로그인 화면은 Selenium으로 자동화하지 말고 공식 인증 라이브러리를 사용하세요.
- 안정적인 `id`, `name`, `data-*` 속성과 의미 기반 CSS selector를 우선 사용하세요.
- 모든 대기에는 타임아웃을 두고, 재시도 횟수에 상한을 설정하세요.
- 고정 `sleep()`보다 `WebDriverWait`와 상태 기반 expected condition을 사용하세요.
- 암시적 대기와 명시적 대기를 혼용하지 마세요.
- 무한 스크롤과 페이지 순회에는 최대 페이지, 최대 건수, 전체 타임아웃과 종료 조건을 설정하세요.
- 크롤링 결과는 필수 컬럼, 데이터형, 중복 키, 빈 값 허용 여부를 검증한 후 저장하세요.
- 브라우저 프로필, 쿠키, 세션 정보는 소스 저장소와 배포물에 포함하지 마세요.

## Google OAuth 2.0 원칙

- Google 인증은 공식 Google 인증 라이브러리와 플랫폼에 맞는 OAuth 클라이언트를 사용하세요.
- 데스크톱 앱 인증에는 시스템 브라우저 기반 승인 흐름과 PKCE를 우선 사용하세요.
- OAuth 사용자 인증, 서비스 계정, 도메인 전체 위임 중 어떤 방식을 사용할지 임의로 결정하지 말고 업무 권한과 배포 환경을 확인하세요.
- 업무에 필요한 최소 scope만 요청하고, scope 확대는 사용자 확인 후 진행하세요.
- `credentials.json`, `client_secret*.json`, `token.json`, 서비스 계정 키를 코드에 하드코딩 하지 마세요.
- 토큰, 인증 코드, 쿠키, 인증 헤더를 로그, 예외 메시지, 화면 캡처 또는 메일에 출력하지 마세요.
- 토큰 만료, 폐기, 권한 철회를 처리하고 필요한 경우 안전한 재인증 흐름을 제공하세요.

## Google Sheets API 원칙

- 쓰기 전에 스프레드시트 ID, 시트명, 대상 범위, 헤더 구조와 사용자 권한을 검증하세요.
- 셀 단위 반복 호출보다 batch 요청을 우선 사용하세요.
- 동일 데이터를 구분할 업무 키를 정의하고, 재실행 시 중복 행이 생성되지 않도록 하세요.
- 실제 쓰기를 수행하는 테스트는 테스트용 스프레드시트에서만 실행하세요.
- 가능하면 dry-run을 제공하고 변경될 범위와 행 수를 먼저 로그로 남기세요.
- 429 및 일시적인 5xx 오류에만 jitter를 포함한 지수 백오프를 적용하세요.
- HTTP 상태 코드만으로 재시도 여부를 결정하지 말고 Google API 오류 사유를 함께 확인하세요.
- 401은 토큰 갱신을 한 번 시도한 뒤 재호출하고, 계속 실패하면 재인증이 필요함을 기록하세요.
- 403 중 권한 부족은 재시도하지 않으며, quota 또는 rate-limit 오류는 제한된 지수 백오프를 적용하세요.
- 400과 데이터 검증 오류는 재시도하지 말고 입력값과 요청 범위를 기록하세요.
- 모든 재시도에는 횟수, 최대 대기 시간과 전체 실행 타임아웃을 설정하세요.
- 부분 성공 여부를 기록하여 재실행 시 성공한 행을 다시 쓰지 않도록 하세요.

## 인증 파일 관리 원칙

- OAuth 클라이언트 설정 파일은 프로젝트 루트의 `Auth/` 폴더에서 관리하세요.
- 액세스 토큰과 리프레시 토큰은 OS 보안 저장소 또는 암호화된 `Auth/` 외부 저장소에 보관하세요.
- `Auth/` 폴더는 PyInstaller `datas` 및 실행 파일 배포물에 포함하지 마세요.
- 인증 파일의 Git 커밋 여부는 사용자가 직접 확인하고 결정하며, 에이전트는 임의로 추가하거나 스테이징하지 마세요.
- 인증 파일의 내용, 토큰 및 개인 계정 정보는 로그, 응답, 테스트 출력에 노출하지 마세요.
- 패키징 실행 환경에서는 `_MEIPASS`가 아닌 실행 파일 외부의 `Auth/` 경로를 사용하세요.

## 코딩 스타일

- 모든 파일에서 UTF-8 인코딩을 유지하세요.
- 한국어 주석과 명확한 구역 주석을 사용할 수 있습니다.
- 새 함수/클래스는 역할이 분명한 이름을 사용하고, 주석을 작성하며 불필요한 추상화는 추가하지 마세요.
- 예외는 삼키지 말고 로그에 충분한 맥락을 남기세요. 무인 실행에서는 실패 메일 흐름이 깨지지 않아야 합니다.
- 로그는 가능하면 `Common.log.Log`의 `log()` 메서드를 사용하세요.
- 메일 발송은 `Common.email_sender.EmailSender`를 사용하되, 테스트 중 실제 메일이 발송되지 않도록 주의하세요.
- GUI 스레드에서 오래 걸리는 작업을 직접 실행하지 말고, 기존 `Worker(QThread)` 패턴을 따르세요.

## 프로젝트 구조
```
CAU000/
├── Auth/                      # OAuth 설정 및 인증 파일
│
├── Common/                    # 설정, 메일, 로그 등 공통 모듈
│   ├── config.py              # ini 설정
│   ├── email_sender.py        # 메일 발송
│   └── log.py                 # 공통 로그
│
├── Function/                  # 애플리케이션 및 업무 유틸리티
│   ├── app_logger.py          # 실행 환경별 로깅 설정
│   ├── app_ui.py              # 플랫폼별 GUI·아이콘 경로 및 공통 GUI
│   ├── app_ui_logger.py       # QTextBrowser 로깅 설정
│   ├── file_util.py           # 파일 유틸리티
│   └── knw_agit.py            # Agit 업무 자동화
│
├── Resource/                  # GUI 및 애플리케이션 리소스
│   ├── checkmark.svg          # UI에서 사용하는 체크 표시 리소스
│   ├── icon.icns              # macOS 애플리케이션 아이콘
│   ├── icon.ico               # Windows 애플리케이션 아이콘
│   ├── main_macos.ui          # macOS용 Qt Designer GUI
│   └── main_windows.ui        # Windows용 Qt Designer GUI
│
├── Service/                   # 업무 전역 상수 및 서비스
│   └── constants.py           # 프로세스 정보와 앱 버전 상수
│
├── Setup/                     # UI 변환·빌드·버전 정보 설정 도구
│   ├── build.bat              # Windows PyArmor·PyInstaller 빌드 실행 스크립트
│   ├── compile_ui.py          # UI를 PySide6 모듈로 변환
│   └── version_info.rc        # Windows .exe 버전 정보
│
├── .gitignore                 # Git 무시 파일 설정
├── AGENTS.md                  # AI 에이전트 작업 지침
├── main.py                    # PySide6 GUI 실행 진입점
├── main_macos.spec            # macOS .app PyInstaller 설정
├── main_unattended.py         # 무인·스케줄러 실행 진입점
├── main_windows.spec          # Windows .exe PyInstaller 설정
├── README.md                  # 프로젝트 설명
└── requirements.txt           # Python 의존성 목록
```
